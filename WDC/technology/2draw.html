Web端数据可视化的研究和应用

任何一个Web端的图形、图像应用，在技术选型时，必然会首先面临SVG和Canvas的选择问题，本节就是介绍在Web可视化方向上，SVG和Canvas的优劣。再开始阅读本章前，希望各位读者能精确掌握，图形、图像、位图、矢量图的概念。此处可参阅[12]

4.2.1 SVG 可伸缩矢量图形标记语言

SVG学习成本低。SVG 是一种使用 XML 描述 2D 图形的语言。而且SVG 基于 XML，这意味着 SVG DOM 中的每个元素都是可用的。这意味着SVG从表层的语言设计层面，可以看作是不在同一命名空间下的HTML。而HMTL是基本上每个Web程序员最熟悉、最先掌握的标记语言。所以在使用SVG时，我们能沿用以往我们熟悉的DOM操作的概念。由此可大大降低SVG的学习成本。

SVG更易被操作。SVG有着XML的结构，而且我们可以为某个元素附加 JavaScript 事件处理器。在 SVG 中，每个被绘制的图形均被视为对象。如果 SVG 对象的属性发生变化，那么浏览器能够自动重现图形。所以从编程角度讲SVG很容易被操作，这点也是SVG被很多图形类库选择的最大原因。

不依赖分辨率。SVG生成的是彻头彻尾的矢量图形，所以SVG图形无需担心屏幕分辨率的问题，这点在屏幕分辨率千变万化的移动互联网时代显得尤为重要。

4.2.2 Canvas HTML5新标签

Canvas 是伴随着HTML5热潮最重要的角色之一，它自己没有行为，但却把一个绘图 API 展现给客户端 JavaScript 以使脚本能够把想绘制的东西都绘制到一块画布上。Canvas 在操作和编程上基本没什么太多可称道的优点，但Canvas先天是基于像素的技术，所以它在处理大数据对象时具有先天的优势。

4.2.3 Canvas vs SVG 总结

Canvas 和 SVG 的对比以及该如何选择，应该来说在网上有大量的文章在讨论，有的偏向一方，有的说各有利弊。其中，微软Developer Network 的一片文章[13]非常精准、详细、客观的从技术和实验的角度描述了两者的异同。这里笔者只能从自己的主观的角度谈谈在Web可视化领域对SVG和Canvas技术选型的看法。

总的来讲，就像位图和矢量图都有存在的理由一样，SVG和Canvas也都有各自存在的理由，并没有说谁能替代谁。对于可视化来讲，从开发成本和推广的角度，SVG是更加好的选择。但如果涉及大数据处理，要从性能上优化的时候，则有些场景必须考虑Canvas。短期来看，SVG的易用性使得其在未来几年内的发展速度会超过Canvas，但从技术的角度出发，长期的发展方向则是Canvas。

在Web可视化这块，虽然数据是大数据，但是从体验的角度出发，设计师一般不会把可视化图形设计成过于密集的图形。一般会通过聚合、拆分的手段使得一个图表表现的数据小于10万。Canvas比SVG最主要的优势就在于对大数量对象处理的性能上，但在这个量级下，Canvas和SVG的性能几乎相差无几。再加上SVG在操作上的优势，所以我认为对于简单的2D Web数据可视化领域，SVG是更佳的技术选型。
